## 一、现状诊断

### 知识库资产盘点

| 维度 | 现状 |
|------|------|
| 文档数量 | 18 篇结构化 Markdown |
| 业务域 | 6 个（平台、支付还款、普惠、海外、消金、风控） |
| 单篇结构 | YAML frontmatter → L3 契约 → 业务正文（目标/用户/指标/页面识别表/黄金流程/经验沉淀） |
| 视觉资产 | COS 托管截图 + Figma 链接 + Gemini 视觉描述 |
| 来源 | 飞书 Wiki 原文提炼 |

### 核心矛盾

知识库已具备 **"AI-ready"** 的文档结构，但从"被动文档"到"主动认知"之间，存在三个断层：

1. **发现层缺失** — Agent 接到任务后，不知道该读哪篇文档。18 篇全塞进去会爆 context window
2. **消费协议缺失** — 读了之后怎么用？约束什么？参考什么？复用什么？没有统一规则
3. **工作流缺失** — 从"接到需求"到"产出方案"的步骤编排不存在

## 二、业界调研核心发现

### 2.1 三个关键模式

#### 模式一：Shopify Polaris 式 — 知识分层暴露

设计系统分成 **Token → 组件 → 模式（Pattern）** 三层。Agent 不是"搜索文档"，而是按层级精确定位。Shopify 做了 MCP Server 让任何 Agent 直接查询最新组件文档。

**启示：** 知识应该有层级，Agent 应该能按粒度访问。

#### 模式二：Canva Orchestrator 式 — 意图路由 + 上下文注入

请求进来后先做 **意图分类**（你要干嘛？），再 **注入品牌知识库**（你的约束是什么），最后路由到合适的执行模块。

**启示：** 先窄化再注入，不是把所有知识都塞给模型。

#### 模式三：Figma Agent 式 — 四层结构化

Design Layer（布局）、Token Layer（变量）、Logic Layer（交互规则）、Knowledge Layer（文档约束）。AI 按需读取对应层。

**启示：** 同一份知识，不同场景需要不同的"视角"。

### 2.2 共性结论

> 设计知识库的 AI 化不是"把文档喂给模型"，而是 **重构知识的消费接口**。

- Shopify 通过 MCP Server 暴露 Polaris
- Figma 通过四层结构化让 Agent 读懂设计系统
- Canva 通过 Brand Kit 注入实现品牌一致性
- OpenAI GPTs 用知识文件 + RAG 动态检索注入 context

### 2.3 技术选型参考

| 平台/方案 | 知识注入机制 | 适用场景 |
|-----------|-------------|---------|
| Claude Code Skill | System prompt + 文件读取 + 工具调用 | 本地开发工作流，我们选择的路径 |
| MCP Server | JSON-RPC 2.0 协议暴露工具/资源 | 跨 Agent 通用访问 |
| OpenAI GPTs | 文件上传 + RAG | 对话式问答 |
| RAG（向量检索） | Embedding + 相似度匹配 | 大规模非结构化文档 |

**我们的选择：Claude Code Skill**
- 团队已在用 Claude Code
- Skill 机制天然支持文件读取和工具编排
- 不需要额外基础设施（向量库、MCP 服务等）

## 三、产出约束与迭代循环（第二轮补充）

### 3.0 产出约束

| 约束项 | 规格 |
|--------|------|
| 平台 | iOS 移动端 |
| 画布尺寸 | 375 × 812 pt |
| 高度规则 | 自适应（根据内容延伸） |
| 产出形态 | UI 设计页面（可交付给开发的设计稿） |

### 3.1 核心缺失：页面清单机制

**第一轮架构的致命问题：** 假设 Agent 一次性产出完整方案。但真实设计工作的本质是——

> 人在使用 Agent，通过 Skill 工作流，**渐进式**确定最终的页面清单和每一页的设计。

页面清单 = 实际生产工作中需要做的页面列表。它对应的是工作量、排期、交付物。这不是 AI 能一次性决定的，而是人机协同的迭代结果。

### 3.2 业界迭代机制调研（第二轮新增）

#### Obsidian 的迭代循环

Obsidian 生态通过三个机制实现知识的渐进精炼：

1. **Execution Loop**（执行循环）：capture → connect → decide → ship。不是"写完就存"，而是"写了之后不断连接、决策、产出"
2. **Karpathy LLM Wiki 模式**：Ingest → Compile → Query → Lint → Rebuild。AI 不是搜索知识，而是持续"编译"知识——每次查询都强化知识库
3. **Dataview + Kanban + ProgressTracker**：用结构化查询驱动状态追踪，每个交付物映射为一张卡片，进度条自动更新

#### LangGraph 状态机模式

LangGraph 解决了"长流程中人机协同"的核心问题：

1. **interrupt() + Command**：在关键节点暂停执行、保存状态、展示信息给人、等待人的决策、恢复执行
2. **Checkpointer**：每一步都保存快照，中断后可以从任意检查点恢复
3. **Time-Travel**：可以回退到任何历史检查点，修改状态，重新执行——允许设计师"反悔"

#### Agent 设计模式

2026 年的 Agent 设计模式中，与我们场景最匹配的：

1. **Generator-Critic-Refiner**：生成草案 → 评审 → 精炼 → 循环直到达标
2. **Iterative Refinement Pattern**：每个循环修改存储在 session state 中的结果，渐进提升
3. **Sequential Pipeline + Human Checkpoint**：线性依赖 + 人工审核门

## 四、以终为始：汇报模板驱动工作流（第三轮补充）

### 4.1 核心思路

> 最终产出 = 一份给领导的完整设计方案汇报。定义好汇报模板的每个板块（槽位），再反推每个工作流节点需要产出什么来填满它。像堆乐高——先看完成图，再逐块拼装。

模板文件：`设计方案汇报模板.html`

### 4.2 汇报模板 = 9 个槽位

| 槽位 | 板块名称 | 来源阶段 | 填充内容 |
|------|----------|---------|---------|
| **A** | 需求背景 | P1 需求理解 | 项目发起原因、业务痛点、用户反馈 |
| **B** | 产品目标 | P1 需求理解 | 业务目标 + 衡量方式 + 目标值 |
| **C** | 目标用户 | P1 需求理解 | 用户群体特征，从知识库提取 + 新需求调整 |
| **D** | 关键指标 | P1 需求理解 | 数据指标名称、定义、基线、预期 |
| **E** | 设计流程 | P2 页面清单 | 主链路（成功路径）+ 分支/异常路径 |
| **F** | 页面清单 | P2 页面清单 | 全部页面列表（名称、新增/复用、参考来源、状态）= 工作量 |
| **G** | 设计方案 | P3 逐页设计 | 每页 375×812 UI 设计稿，逐页循环完成 |
| **H** | 设计自检 | P4 整体评审 | 流程完整性、业务规则一致性、UI 规范、信息不足项 |
| **I** | 知识库命中 | 自动（全程） | 引用了哪些知识库文档、命中关键词、引用位置 |

### 4.3 工作流节点 → 槽位的映射

```
P1 需求理解 + 知识检索
├── 填充 → Slot A 需求背景
├── 填充 → Slot B 产品目标
├── 填充 → Slot C 目标用户
├── 填充 → Slot D 关键指标
└── 触发 → Slot I 知识库命中（自动记录命中了哪些文档）

    ★ 人工检查点：确认理解准确

P2 页面清单生成
├── 填充 → Slot E 设计流程（主链路 + 分支）
├── 填充 → Slot F 页面清单（= 工作量确认）
└── 追加 → Slot I 知识库命中

    ★ 人工检查点：调整页面清单，确认工作量

P3 逐页设计（循环）
├── 逐页填充 → Slot G 设计方案
│   └── 每页：Agent 生成 375×812 → 人工审核 → 通过/精炼
└── 追加 → Slot I 知识库命中

    ★ 每页一个检查点

P4 整体评审
├── 填充 → Slot H 设计自检
└── 最终确认 → Slot I 知识库命中完整性

    ★ 通过 = 方案完成 / 不通过 = 回到 P3 修改
```

### 4.4 乐高拼装的关键特性

1. **独立填充**：每个槽位可以独立完成，P1 的四个槽位完成后不需要等 P3
2. **渐进可见**：每填完一个槽位，汇报模板就多一块内容，随时可以预览当前进度
3. **可回退**：如果 P4 评审发现问题，可以只回到特定槽位重新填充，不影响其他已完成的
4. **来源可追溯**：每个槽位标注了来源阶段，审阅者知道每块内容是怎么来的
5. **状态可视化**：虚线边框 = 待填充，实线边框 = 已完成

## 五、架构方案：三层 Skill 体系

### 3.1 整体架构

```
┌───────────────────────────────────────────────────┐
│              用户入口（3 个 Skill）                  │
│                                                     │
│  /biz-solution    基于知识库产出设计方案              │
│  /biz-review      基于知识库评审现有方案              │
│  /biz-analyze     基于知识库理解和拆解需求            │
│                                                     │
└─────────────────────┬───────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────┐
│              知识路由层（manifest.json）               │
│                                                       │
│  职责：意图 → 关键词匹配 → 返回命中文档路径列表         │
│  特点：几百 token，不吃 context                        │
│  机制：规则式精确路由，不是 RAG 向量检索               │
│                                                       │
└─────────────────────┬───────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────┐
│              知识消费层（按需读取）                     │
│                                                       │
│  只读命中的 1-3 篇文档                                │
│  按场景提取不同部分：                                  │
│                                                       │
│    /biz-solution → 页面识别表 + 黄金流程 + 视觉证据    │
│    /biz-review   → 业务规则 + 异常路径 + 关键约束      │
│    /biz-analyze  → 项目目标 + 指标 + 关联流程          │
│                                                       │
└───────────────────────────────────────────────────────┘
```

### 4.2 三个 Skill 的职责定义

#### /biz-solution — 方案产出

| 维度 | 说明 |
|------|------|
| 触发场景 | 设计师接到新需求，需要产出设计方案 |
| 输入 | 需求描述（文字/PRD/口头描述） |
| 知识消费 | 页面识别表（复用已有页面）、黄金流程（参考路径）、视觉证据（设计参考） |
| 输出 | 页面结构建议、交互流程、组件复用清单、Figma 参考链接 |
| 核心价值 | 不从零设计，而是站在已有业务知识上设计 |

#### /biz-review — 方案评审

| 维度 | 说明 |
|------|------|
| 触发场景 | 方案完成后，需要对照业务规则自检 |
| 输入 | 设计方案（截图/描述/Figma链接） |
| 知识消费 | 业务规则、异常路径、关键约束、"不能混淆的概念" |
| 输出 | 一致性检查报告：哪些符合/哪些偏离/哪些缺失 |
| 核心价值 | 用知识库中的"经验沉淀"防止重复踩坑 |

#### /biz-analyze — 需求拆解

| 维度 | 说明 |
|------|------|
| 触发场景 | 拿到需求描述后，需要理解业务背景和上下文 |
| 输入 | 需求描述/PRD片段 |
| 知识消费 | 项目目标、目标用户、数据指标、关联流程 |
| 输出 | 业务上下文分析、关联流程梳理、任务拆解清单 |
| 核心价值 | 新人或跨域设计师能快速理解业务全貌 |

### 4.3 关键设计决策

| 决策点 | 选择 | 原因 |
|--------|------|------|
| 知识发现 | manifest.json（结构化索引） | 比全文搜索快，不吃 context |
| 知识粒度 | 按文档整篇读取 | 每篇文档本身就是完整业务域，切片会丢上下文 |
| Skill 数量 | 3 个场景 Skill + 1 个路由模块 | 针对性 prompt 优化，避免万能 Skill 泛化 |
| 视觉证据 | URL 引用而非嵌入 | 截图已在 COS 上，Agent 可按需 fetch |
| 检索方式 | 规则路由而非向量检索 | 18 篇文档用 RAG 是杀鸡用牛刀 |

### 5.2 路由匹配规则

Agent 收到用户需求后的匹配流程：

1. 读取 manifest.json（约 200-400 token）
2. 将用户需求与每篇文档的 `keywords` + `match` + `summary` 进行语义匹配
3. 返回 top 1-3 命中文档的 `path`
4. 按场景读取文档对应部分

### 7.2 三个 Skill 的核心设计

四阶段状态机，每阶段结束有人工检查点：

状态通过 `.biz-session.json` 持久化，支持中断恢复和回退。

#### /biz-review — 独立评审

一次性评审流程：知识路由 → 提取评审基准（业务规则/异常路径/经验沉淀）→ 逐项检查 → 输出报告。
评审维度：流程完整性、业务规则一致性、经验沉淀对照、UI 规范、信息缺口。

#### /biz-analyze — 独立分析

一次性分析流程：知识路由 → 提取上下文（目标/用户/指标/关联流程）→ 输出分析报告。
产出：业务定位、关联流程梳理、已有页面资产盘点、任务拆解、信息缺口。

### 7.3 共享协议

三个 Skill 均遵循：
1. 先读 `manifest.json` 做知识路由
2. 按需读取 1-3 篇命中文档
3. 输出中标注知识库命中证据
