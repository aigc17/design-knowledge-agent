# /biz-review — 设计方案评审

> 基于业务知识库，对照业务规则评审现有设计方案

## 评审对象

$ARGUMENTS

---

## 执行协议

### 1. 知识路由

读取 `业务知识库/manifest.json`，从评审对象描述中提取关键词，匹配命中 1-3 篇知识文档，读取完整内容。

### 2. 提取评审基准

从命中文档中提取：
- **业务规则**：核心流程约束、状态流转规则、数据校验规则
- **异常路径**：错误处理、边界情况、降级方案
- **经验沉淀**：历史踩坑点、"不能混淆的概念"、设计禁区
- **关键约束**：合规要求、技术限制、业务红线

### 3. 逐项评审

对评审对象进行以下维度检查：

```
## 评审报告

### 流程完整性
- [x/☐] 主链路覆盖：{说明}
- [x/☐] 异常路径覆盖：{说明}
- [x/☐] 关键节点不遗漏：{说明}

### 业务规则一致性
- [x/☐] {规则1}：{符合/偏离，具体说明}
- [x/☐] {规则2}：{符合/偏离，具体说明}

### 经验沉淀对照
- [x/☐] {历史踩坑点1}：{是否避免}
- [x/☐] {历史踩坑点2}：{是否避免}

### UI 规范
- [x/☐] 页面尺寸 375×812 合规
- [x/☐] 组件复用正确
- [x/☐] 视觉一致性

### 信息缺口
- {知识库中缺失但方案需要的信息}
- {需要人工补充确认的部分}

### 知识库命中
- 命中：{文档标题} — 关键词：{匹配词}
- 引用章节：{具体引用的知识库章节}
```

### 4. 输出结论

在报告末尾给出总结：
- **通过项**：N 项
- **偏离项**：N 项（列出具体偏离及建议修改）
- **缺失项**：N 项（列出遗漏内容及补充建议）
- **整体建议**：通过 / 需修改后再审 / 需重新设计

## 核心原则

1. **知识库是裁判** — 评审标准来自知识库沉淀，不是主观判断
2. **精确引用** — 每条评审意见标注知识库来源章节
3. **建设性反馈** — 指出问题的同时给出修改方向，引用知识库中的正确做法
4. **区分硬伤和优化** — 业务规则冲突是硬伤必须修改，UI 细节是优化可酌情处理
