# /biz-solution — 设计方案产出

> 基于业务知识库，产出一份可交付的设计方案汇报。最终产物 = 填满 9 个槽位的汇报 HTML。

## 需求输入

$ARGUMENTS

---

## 启动协议

1. **读取路由索引** — `业务知识库/manifest.json`
2. **读取汇报模板** — `设计方案汇报模板.html`（理解 9 个槽位结构，这是最终产物的骨架）
3. **检查会话状态** — `.biz-session.json`
   - 存在 → 读取，从 `phase` 指示的阶段继续
   - 不存在且有需求输入 → 新建会话，进入 P1
   - 不存在且无输入 → 提示用户输入需求描述
4. **知识路由** — 用需求关键词匹配 manifest 中 `documents[].keywords` + `match` + `summary`，选中 1-3 篇最相关文档，读取完整内容
5. **创建输出目录** — `业务知识库/output/`

---

## 来源标注规则

每条写入槽位的内容必须标注来源：

- **📗 KB** = 从知识库文档中提取或直接推导（标注具体文档+章节）
- **🔮 推理** = Agent 基于设计经验或通用知识推理（KB 中无对应内容）

Slot I 自动汇总：哪些槽位的哪些内容来自哪篇 KB 文档。如果一个槽位全部是 🔮 推理，说明 KB 对该部分无覆盖——这本身就是有价值的信息。

---

## P1 · 需求理解 → 填充 Slot A/B/C/D

**读取**：命中文档中的「项目目标」「目标用户特征」「数据指标」章节

**填充：**
- **Slot A 需求背景** — 项目发起原因、业务痛点（每条标注 📗/🔮）
- **Slot B 产品目标** — 目标 + 衡量方式 + 目标值（每条标注 📗/🔮）
- **Slot C 目标用户** — 用户群体特征（每条标注 📗/🔮）
- **Slot D 关键指标** — 指标名称、定义、基线、预期（每条标注 📗/🔮）
- **Slot I 追加** — 记录本阶段命中的 KB 文档和引用章节

**产出**：生成汇报 HTML → `业务知识库/output/{项目简称}-report.html`
此时 Slot A-D 已填充（实线边框），Slot E-H 仍为占位符（虚线边框）。在浏览器打开供用户预览。

**保存状态**：`.biz-session.json`，`phase: "understanding"`

**★ 检查点** — 打开汇报 HTML，问用户：

> 汇报的 Part 1（需求认知）已填充，请在浏览器查看。理解是否准确？确认后进入 P2。

---

## P2 · 页面清单 → 填充 Slot E/F

**读取**：命中文档中的「页面识别总表」「黄金流程」章节

**填充：**
- **Slot E 设计流程** — 主链路 + 分支/异常路径，节点标注新增/复用（每条标注 📗/🔮）
- **Slot F 页面清单** — 表格：页面名称、类型、参考来源、状态（每条标注 📗/🔮）
- **Slot I 追加** — 本阶段新增的 KB 引用

**产出**：更新汇报 HTML，Slot A-F 已填充，Slot G-H 仍占位。浏览器打开。

**保存状态**：`phase: "inventory"`，`inventory[]` 写入每页信息

**★ 检查点** — 打开汇报 HTML，问用户：

> Part 2（设计规划）已填充。页面清单是否需要调整？确认后进入 P3 设计全部页面。

---

## P3 · 逐页设计 → 填充 Slot G

**对 `inventory` 中所有页面一次性设计**（不逐页确认，全部完成后统一审核）：

**每页设计流程：**
1. 读取该页参考来源在知识库中的视觉描述、Figma 链接、交互规则
2. 生成 iOS 375×812 pt UI 设计
3. 记录设计决策（每条标注 📗/🔮：哪些来自 KB 参考页面，哪些是新设计）

**填充：**
- **Slot G 设计方案** — 所有页面的 375×812 设计稿，嵌入汇报 HTML 中作为手机框展示。每页附设计说明 + 知识库依据。
- **Slot I 追加** — 本阶段引用的 KB 页面视觉描述和设计规则

**产出**：更新汇报 HTML，Slot A-G 已填充，仅 Slot H 仍占位。浏览器打开。

**保存状态**：所有页面 `status: "completed"`，`phase: "page-design"`

**★ 检查点** — 打开汇报 HTML，问用户：

> Part 3（设计方案）全部页面已完成。请在浏览器查看。是否需要修改？确认后进入 P4 整体评审。

---

## P4 · 整体评审 → 填充 Slot H，定稿 Slot I

**读取**：重新读取命中文档，提取业务规则、异常路径、关键约束

**填充：**
- **Slot H 设计自检** — 四个维度逐项检查：
  - 流程完整性（主链路 + 异常路径覆盖）
  - 业务规则一致性（每条规则标注 📗 来源）
  - UI 规范合规（尺寸、组件、视觉一致性）
  - 信息不足项（KB 缺失的信息，需人工补充的部分）
- **Slot I 定稿** — 汇总全部 KB 引用，每条标注：文档标题、引用章节、用在哪个 Slot、贡献类型（驱动设计 / 约束校验 / 视觉参考）

**产出**：最终版汇报 HTML，9 个槽位全部填满。浏览器打开。

**★ 检查点** — 打开汇报 HTML，问用户：

> 汇报已完成，9 个槽位全部填满。是否通过？
> - 通过 → `phase: "completed"`
> - 不通过 → 指定修改内容，我更新对应槽位

## 产出约束

| 约束 | 规格 |
|------|------|
| 最终产物 | `业务知识库/output/{项目简称}-report.html` — 填满的汇报模板 |
| 模板结构 | `设计方案汇报模板.html`（9 槽位 × 4 阶段映射） |
| 设计尺寸 | iOS 375 × 812 pt（宽固定，高自适应） |
| 来源标注 | 每条内容标注 📗 KB / 🔮 推理 |

## 会话状态结构

```json
{
  "sessionId": "ses_{YYYYMMDD}_{shortId}",
  "phase": "understanding | inventory | page-design | review | completed",
  "requirement": "需求原文",
  "kbHits": ["文档路径"],
  "reportFile": "output/{项目简称}-report.html",
  "inventory": [
    { "id": "p1", "name": "页面名", "type": "new|reuse", "status": "pending|completed", "reference": "参考来源" }
  ],
  "decisions": ["设计决策"],
  "kbAudit": [
    { "slot": "A", "source": "kb", "doc": "文档标题", "section": "章节名", "type": "驱动设计|约束校验|视觉参考" },
    { "slot": "A", "source": "inferred", "reason": "KB未覆盖退款场景" }
  ]
}
```

## 核心原则

1. **最终产物是汇报 HTML，不是聊天消息** — 每个阶段的价值体现在模板槽位被填满
2. **来源透明** — 📗/🔮 标注让 KB 贡献度一目了然，不装饰、不注水
3. **渐进可见** — 每完成一个阶段，汇报 HTML 更新一次，浏览器可实时预览进度
4. **可回退** — 用户随时指定槽位重新填充，不影响其他已完成的槽位
