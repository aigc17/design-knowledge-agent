# /biz-solution — 设计方案产出

> 四阶段人机协同工作流：需求理解 → 页面清单 → 逐页设计 → 整体评审

## 需求输入

$ARGUMENTS

---

## 启动协议

按顺序执行：

1. **读取路由索引** — `业务知识库/manifest.json`
2. **检查会话状态** — 当前工作目录下 `.biz-session.json`
   - 存在 → 读取，从 `phase` 指示的阶段继续
   - 不存在且有 `$ARGUMENTS` → 新建会话，进入 P1
   - 不存在且无输入 → 提示用户输入需求描述
3. **知识路由** — 用需求关键词匹配 manifest 中 `documents[].keywords` + `match` + `summary`，选中 1-3 篇最相关文档，读取完整内容

---

## P1 · 需求理解

**读取**：命中文档中的「项目目标」「目标用户特征」「数据指标」「业务规则」章节

**产出**（对应汇报模板 Slot A-D + I）：

```
## 需求背景（Slot A）
{项目发起原因、业务痛点、用户反馈}

## 产品目标（Slot B）
| 目标 | 衡量方式 | 目标值 |
|------|---------|--------|
| ...  | ...     | ...    |

## 目标用户（Slot C）
{用户群体特征，从知识库提取 + 新需求调整}

## 关键指标（Slot D）
| 指标 | 定义 | 基线 | 预期 |
|------|------|------|------|
| ...  | ...  | ...  | ...  |

## 知识库命中（Slot I）
- 命中：{文档标题} — 关键词：{匹配的关键词}
```

**保存状态**：写入 `.biz-session.json`，`phase: "understanding"`

**★ 检查点** — 停下来问用户：

> 以上业务理解是否准确？需要补充或修正什么？确认后我进入 P2 生成页面清单。

---

## P2 · 页面清单

**读取**：命中文档中的「页面识别总表」「黄金流程」章节

**产出**（对应 Slot E-F）：

```
## 设计流程（Slot E）

主链路：{页面A} → {页面B} → {页面C} → ...

分支/异常路径：
- {条件} → {页面X}
- {条件} → {页面Y}

## 页面清单（Slot F）

| # | 页面名称 | 类型 | 参考来源（知识库） | 状态 |
|---|---------|------|--------------------|------|
| 1 | xxx     | 新增 | —                  | pending |
| 2 | xxx     | 复用调整 | 参考「{知识库页面名}」| pending |

合计 N 页 · 新增 n1 · 复用调整 n2
```

**保存状态**：`phase: "inventory"`，将每页写入 `inventory[]`

**★ 检查点** — 停下来问用户：

> 页面清单是否需要调整？可以增删页面、调整顺序、修改类型。确认后我逐页开始设计。

---

## P3 · 逐页设计（循环）

对 `inventory` 中下一个 `status: "pending"` 的页面执行：

**读取**：该页参考来源在知识库中的视觉描述、Figma 链接、交互规则

**产出**（对应 Slot G 的一张卡片）：

1. 使用 Pencil MCP 工具创建 iOS 375×812 pt 的 UI 设计页面（.pen 文件）
2. 设计基于知识库中的参考页面样式，融入新需求调整
3. 输出设计说明：页面名称、核心交互、与参考页面的差异点

**保存状态**：该页 `status: "in-progress"`

**★ 检查点** — 停下来问用户：

> {页面名} 的设计是否通过？
> - 通过 → 标记完成，继续下一页
> - 修改 → 请说明修改意见，我来精炼

用户确认通过 → `status: "completed"` → 自动进入下一个 pending 页面
全部 completed → 进入 P4

---

## P4 · 整体评审

**读取**：重新读取全部命中文档，提取业务规则、异常路径、关键约束

**产出**（对应 Slot H）：

```
## 设计自检（Slot H）

- [x] **流程完整性**：{主链路完整/异常路径覆盖情况}
- [x] **业务规则一致性**：{与知识库规则的符合/偏离}
- [ ] **UI 规范合规**：{尺寸/组件/视觉一致性}
- [ ] **信息不足项**：{知识库缺失的信息，需人工补充}
```

**保存状态**：`phase: "review"`

**★ 检查点** — 停下来问用户：

> 评审结果如上。是否通过？
> - 通过 → 方案完成，`phase: "completed"`
> - 不通过 → 请指定需要修改的页面，回到 P3 精炼

## 产出约束

| 约束 | 规格 |
|------|------|
| 平台 | iOS 移动端 |
| 尺寸 | 375 × 812 pt（宽固定，高自适应） |
| 设计工具 | Pencil MCP（.pen 文件） |
| 汇报模板 | `设计方案汇报模板.html`（9 槽位） |

## 会话状态结构

`.biz-session.json` 位于工作目录下，结构：

```json
{
  "sessionId": "ses_{YYYYMMDD}_{shortId}",
  "phase": "understanding | inventory | page-design | review | completed",
  "requirement": "用户的需求描述原文",
  "kbHits": ["业务知识库/ai/风控/03-风控还借宝流程.md"],
  "inventory": [
    {
      "id": "p1",
      "name": "页面名",
      "type": "new | reuse",
      "status": "pending | in-progress | completed",
      "reference": "参考来源",
      "designFile": "output/p1-xxx.pen"
    }
  ],
  "decisions": ["设计决策记录"],
  "slots": {
    "A": "需求背景内容",
    "B": "产品目标内容",
    "I": [{ "doc": "文档标题", "keywords": ["命中词"], "usedIn": ["A", "E"] }]
  }
}
```

## 核心原则

1. **每个阶段结束必须停下来等用户确认**，不自动跳到下一阶段
2. **知识库是参考不是教条** — 新需求可以覆盖知识库中的旧规则，但要显式标注差异
3. **渐进可见** — 每完成一个槽位，用户就能看到方案在成长
4. **可回退** — 用户随时可以说"回到 P2 调整清单"或"重新设计第 3 页"
