PART 01 项目介绍

这是什么，以及为什么需要它

一套业务知识库驱动的设计方案自动化系统。它让团队沉淀的业务流程知识，从「需要人去翻」的静态文档，升级为「AI 能精准调用」的认知资产。

设计知识库的 AI 化，不是「把文档喂给模型」，而是重构知识的消费接口 —— 让 Agent 用最小代价找到正确知识，并按统一协议消费它。

核心矛盾：从「被动文档」到「主动认知」的三个断层

知识库已经是结构化、AI-ready 的 Markdown，但文档「能被读」不等于「会被用」。中间隔着三道坎。

01

发现层缺失

Agent 接到任务后，不知道该读哪篇文档。18 篇全塞进上下文会直接撑爆 context window。

解法 › manifest.json 路由表，关键词精确命中 1–3 篇
02

消费协议缺失

读了之后怎么用？约束什么、参考什么、复用什么？没有统一规则，每次发挥都不一样。

解法 › 三级来源标注 + 按章节分层读取
03

工作流缺失

从「接到需求」到「产出方案」的步骤编排不存在，无法保证产物形态稳定可交付。

解法 › 4 阶段状态机 + 模板填槽

解法：知识路由驱动的四层架构

💬

用户需求（自然语言）

设计师一句话描述需求，无需指定读哪篇文档。例：「还借宝新增退款确认页」

🚪

Skill 入口 .claude/commands/

三个场景化 Skill：方案产出 / 方案评审 / 需求拆解，各自针对性优化 prompt。

仅 /biz-solution 带状态机，用 .biz-session.json 持久化 4 阶段进度
🧭

知识路由 manifest.json

约 300 token 的路由索引。第一遍关键词精确命中 1–3 篇；第二遍跨域扫描全部页面，找相似截图模式。

规则路由，不是 RAG 向量检索 —— 18 篇文档用向量库是杀鸡用牛刀
📚

知识文档 ai/{域}/*.md

按 sectionSchema 分章节读取，不读全文 —— 18 篇文档含大量视觉描述，一次读全会撑爆上下文。

P1 读目标/用户/指标，P2 读流程/页面，各阶段只取所需章节
📄

HTML 模板填槽 → 设计方案产物

双 Tab 报告（汇报视图 + 工作台视图），每条决策标注三级来源，浏览器实时预览。

output/{简称}-report.html，可直接交付领导与设计师

关键决策：为什么是 Claude Code Skill，而不是 RAG

决策点选择原因
知识发现manifest.json 结构化索引比全文搜索快，几百 token 不吃 context
检索方式规则路由 ✓18 篇文档用向量检索是杀鸡用牛刀，规则命中更精确可控
知识粒度按章节分层读取每篇是完整业务域，按 sectionSchema 取所需章节，避免上下文爆炸
承载平台Claude Code Skill ✓团队已在用，天然支持文件读取 + 工具编排，零额外基础设施
视觉证据URL 引用而非嵌入截图已托管在 COS，Agent 按需 fetch，不污染上下文
PART 02 知识库应用场景

知识库到底在什么时候、怎么起作用

从设计师的真实工作场景出发：拿到需求 → 理解业务 → 产出方案 → 评审自检。知识库在每个环节都提供精确的认知支撑。

③ 路由机制：两遍扫描，精确命中 + 跨域复用

这是知识库「主动认知」的核心 —— 既能找到正脸命中的文档，又能跨业务域借用相似的页面模式与流程原型。

PASS 1业务域精确命中
  • 匹配 keywords + match + summary
  • 选中 1–3 篇最相关文档
  • 按 sectionSchema 只读对应章节
  • 标注 [知识库] 来源（文档 + 章节）
PASS 2跨域相似匹配
  • 扫描全部 pages[] 找相似截图模式
  • 比对 7 种 flowArchetype 流程原型
  • 提取截图 URL + 视觉描述 → 候选池
  • 标注 [跨域参考] 来源（页面 + 链接）

④ 来源可追溯：每条决策都标清「凭什么」

审阅者一眼就能区分：哪些是知识库铁证，哪些是跨域借鉴，哪些是 AI 在知识库空白处的自主推理 —— 决策透明，责任清晰。

[知识库]

精确命中文档，标注来源文档 + 具体章节。最高可信度，业务铁证。

[跨域参考]

跨业务域的相似页面截图 / 流程模式，标注来源文档 + 页面名 + 截图链接。

[推理]

知识库无覆盖，Agent 基于上下文独立推理。明确标记，提醒人工复核。
