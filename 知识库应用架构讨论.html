核心共识

设计知识库的 AI 化不是"把文档喂给模型"，而是重构知识的消费接口。知识的组织方式决定了 Agent 的认知质量。

第一轮架构的致命缺陷

假设 Agent 一次性产出完整方案。但真实设计工作是渐进式、人机协同的迭代过程。页面清单 = 实际生产工作量，不是 AI 能一次性决定的。

业界迭代机制

O

Obsidian 执行循环

Karpathy LLM Wiki：Ingest → Compile → Query → Lint → Rebuild。知识不是"存"的，而是"编译"的——每次查询都强化知识库。

L

LangGraph 状态机

interrupt() + Command：关键节点暂停 → 保存状态 → 展示给人 → 等待决策 → 恢复执行。支持 Time-Travel 回退。

G

Generator-Critic-Refiner

生成草案 → 评审 → 精炼 → 循环直到达标。Reflection 模式可将准确率从 80% 提升至 91%。

页面清单状态机

P1
需求理解 + 知识检索
Agent 读取 manifest → 命中相关知识 → 分析需求背景
★ 人工检查点：理解准确？需要补充？
P2
页面清单生成
基于知识库已有页面 + 新需求 → 提出页面清单草案（名称、流程位置、复用/新增、参考页面）
★ 人工检查点：调整清单——增删页面、确认工作量
P3
逐页设计 (循环)
for each page in confirmed_inventory → 生成 375×812 UI → 审核 → 精炼 → 标记完成 → 下一页
p1 · 退款确认页 completed
p2 · 退款结果页（成功） in-progress ←
p3 · 退款结果页（失败） pending
★ 每页一个检查点：通过 / 修改意见 → 精炼
P4
整体评审
对照知识库业务规则 → 全量一致性检查 → 评审报告
★ 通过 or 回到 P3 修改特定页面

Part 5

manifest.json 设计

路由核心 — 让 Agent 用最小代价找到正确知识

manifest.json
{ "version": "1.0", "domains": ["平台", "支付还款", "普惠", "海外", "消金", "风控"], "documents": [ { "id": "risk-hjb", "title": "风控还借宝流程", "domain": "风控", "path": "ai/风控/03-风控还借宝流程.md", "keywords": ["还借宝", "恢复额度", "超市卡", "赔付"], "match": "需求提到风控/还借宝/恢复额度时命中", "summary": "购买超市卡恢复额度、放款失败赔付权益链路", "pages": ["还借宝购买页", "交易鉴权", "购买结果页", ...], "relatedFlows": ["payment-repay", "consumer-credit"] }, // ... 全部 18 篇文档的路由条目 ] }
设计决策 选择 原因
知识发现 manifest.json 比全文搜索快，约 300 token，不吃 context
知识粒度 整篇文档 每篇本身是完整业务域，切片丢上下文
检索方式 规则路由 非 RAG 18 篇文档用向量检索是杀鸡用牛刀
Skill 数量 3 场景 + 1 路由 针对性 prompt 优化，避免万能 Skill 泛化
视觉证据 URL 引用 截图已在 COS，Agent 按需 fetch
